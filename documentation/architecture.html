01

System Overview

Three layers work together: app.py defines the problem (shifts, staff, demands, rules), solver.py translates it into a CP-SAT model and solves it, and models.py provides the shared domain vocabulary. The solver returns an assignments dict mapping (staff, day, shift) → 0 or 1, which can be chained across roster periods via RosterContext.

02

Domain Model

All domain objects live in models.py. ShiftGroup categorises shifts by behaviour flags. Shift is a template assigned to staff on any day. Staff carries a skill set and a list of permitted shifts. Demand specifies the minimum headcount required during a time window on a given date.

03

Constraint Pipeline

Constraints are applied in order inside generate_roster() before the solver runs. Hard constraints define feasibility; the objective drives fairness.

flowchart TD START([generate_roster called]) --> C1 C1["C1 — One shift per day\nAt most 1 work shift and\n1 rest shift per staff per day"] C4["C4 — Target hours\nExactly target_minutes of work\nper staff over the roster"] C5["C5 — Demand coverage\nAt each checkpoint time,\nheadcount ≥ demand.headcount\nfor eligible staff"] C6["C6 — Permitted shifts\nBlock shifts not in\nstaff.permitted_shifts"] C7["C7 — Pre-assignments\nBlock unassigned rest shifts to 0,\nPin pre-assigned shifts to 1,\nblock other work shifts on that day"] C8["C8 — No demand, no work\nBlock all work shifts on days\nwith no demand"] C9["C9 — ConditionalConstraints\nFor each rule: if trigger\nshiftgroup == trigger_val on day d\nthen enforce shiftgroup ==\nenforce_val on day d+offset"] C10["C10 — Boundary carry-over\nApply ConditionalConstraints\nacross roster boundary using\nRosterContext.carry_shifts"] OBJ["Objective — Weighted minimisation\nmin: max consecutive days × 100\n+ overstaffing spread × 20\n+ night/weekend burden × 10"] SOLVE([CP-SAT Solve]) C1 --> C4 --> C5 --> C6 --> C7 --> C8 --> C9 --> C10 --> OBJ --> SOLVE style C1 fill:#f0f4ff,stroke:#2b6cb0 style C4 fill:#f0f4ff,stroke:#2b6cb0 style C5 fill:#f0f4ff,stroke:#2b6cb0 style C6 fill:#f0f4ff,stroke:#2b6cb0 style C7 fill:#f0f4ff,stroke:#2b6cb0 style C8 fill:#f0f4ff,stroke:#2b6cb0 style C9 fill:#fff5f5,stroke:#c84b31 style C10 fill:#fff5f5,stroke:#c84b31 style OBJ fill:#f0fff4,stroke:#2d6a4f
C1
One Shift Per Day
At most one work shift and one rest shift assigned per staff per day. Prevents double-booking.
C4
Exact Target Hours
Each staff member's total work_time across all days must equal exactly target_minutes (e.g. 4800 min = 80 hrs).
C5
Demand Coverage
At each demand checkpoint, the count of eligible staff (skill-filtered) covering that time must meet headcount.
C6
Permitted Shifts
Staff can only be assigned shifts in their permitted_shifts list. Used to restrict night shifts for specific staff.
C7
Pre-Assignments (Leave)
Unassigned rest shifts are blocked to 0. Pre-assigned shifts are pinned to 1, and all other work shifts on that day are blocked to 0.
C8
No Demand, No Work
Work shifts are blocked on days with no demand, preventing the solver from assigning staff unnecessarily on days with no demand.
C9
ConditionalConstraints
Dynamic if/then rules configurable from outside the solver. Replaces all hardcoded post-shift rest logic.
C10
Boundary Carry-Over
Applies ConditionalConstraints across the roster boundary using RosterContext.carry_shifts as hard constraints (trigger already occurred).
OBJ
Weighted Objective
Minimise: max consecutive days × 100 + overstaffing spread × 20 + night/weekend burden × 10.
04

ConditionalConstraint

A ConditionalConstraint encodes an if/then rule as five parameters. Rules are passed into RosterEngine at construction time and applied to every staff member across every day. Passing an empty list disables all rules entirely.

flowchart LR subgraph RULE["ConditionalConstraint(trigger, trigger_val, offset, enforce, enforce_val)"] direction LR T["trigger\nShiftGroup code\ne.g. 'NSG'\nor '*' = all"] TV["trigger_val\n1 = shift assigned\n0 = not assigned"] O["offset\nday delta\n-1, 0, +1, +2 …"] E["enforce\nShiftGroup code\ne.g. 'DSG'\nor '*' = all"] EV["enforce_val\n0 = block\n1 = force"] T --> TV --> O --> E --> EV end subgraph LOGIC["Solver logic for each staff n, day d"] direction TB L1{"trigger == trigger_val\non day d?"} L2["Set enforce == enforce_val\non day d + offset"] L1 -->|Yes| L2 L1 -->|No| L3[No effect] end RULE --> LOGIC style RULE fill:#fff5f5,stroke:#c84b31,stroke-width:1px style LOGIC fill:#f5f2eb,stroke:#6b7280,stroke-width:1px

Active rules in app.py

trigger="NSG" trigger_val=1 offset=+1 → enforce="*" enforce_val=0 // block all work the day after a night shift
trigger="NSG" trigger_val=1 offset=+2 → enforce="DSG" enforce_val=0 // block DSG 2 days after a night shift
trigger="Leaves" trigger_val=1 offset=-1 → enforce="NSG" enforce_val=0 // block night shift the day before leave
05

Chained Roster Flow

Two consecutive rosters are generated sequentially. build_roster_context() extracts the carry-over state from a completed roster which seeds the next roster's boundary constraints via C10.

sequenceDiagram participant T as app.py participant E as RosterEngine participant CP as CP-SAT Solver participant RC as RosterContext Note over T: ROSTER 1 T->>E: generate_roster(roster_start, demands, leaves) E->>CP: Build model with C1,C4,C5,C6,C7,C8,C9 CP-->>E: result_1 {assignments, consec_days, …} E-->>T: result_1 T->>T: print_roster(result_1) T->>T: print_staff_summary(result_1) Note over T: EXTRACT CARRY-OVER STATE T->>E: build_roster_context(result_1, next_roster_start) E->>E: Scan backwards from last day per staff E->>E: Record last shift group + days_ago E->>E: Record consecutive day count E-->>T: RosterContext {carry_consec, carry_shifts} Note over T: ROSTER 2 T->>E: generate_roster(roster_start, demands, leaves, context) E->>CP: Build model + C10 boundary constraints CP-->>E: result_2 {assignments, consec_days, …} E-->>T: result_2 T->>T: print_roster(result_2) T->>T: print_staff_summary(result_2)
06

Boundary Carry-Over

RosterContext.carry_shifts stores the most recent shift group and how many days ago it was assigned (relative to the next roster's start). C10 then matches this against all active ConditionalConstraint rules and applies them as hard constraints on the first days of the new roster — no OnlyEnforceIf needed since the trigger already occurred.
