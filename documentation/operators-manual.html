1What this system is

RosterEngine builds staff schedules ("rosters") automatically. You describe who is available, what shifts exist, and how many people you need at which hours — and a constraint solver (Google OR-Tools CP-SAT) finds an assignment that satisfies every rule while balancing fairness.

It runs as six Docker containers:

ServicePortWhat it does
frontend5173The web app (React). This is where you work day to day.
backend8000The API (FastAPI). Swagger docs at /docs.
celery_worker—Runs the solver in the background. Roster generation happens here, not in the API.
mcp8001MCP server — lets an AI chatbot drive the roster API (optional, see §8).
postgres5432The database. All data lives in a Docker volume.
redis6379Queue between the API and the worker.

Because solving is asynchronous, generating a roster never blocks the UI: you submit it, the roster sits in running, and the page polls until the worker finishes.

2Starting & stopping

Run everything from the repo root, on the main branch. All generated credentials live in the repo's .env (gitignored).
cd /home/guo/roster-engine

docker compose -f docker-compose.dev.yml up -d      # start everything
docker compose -f docker-compose.dev.yml down       # stop (data survives)
docker compose -f docker-compose.dev.yml down -v    # stop AND WIPE the database
docker compose -f docker-compose.dev.yml logs -f celery_worker   # watch the solver
docker compose -f docker-compose.dev.yml ps         # service health

One-time setup (already done on this machine): image build, database migrations (alembic upgrade head), and first user creation. You only repeat those after down -v.

3Terminology

The app's vocabulary, in the order you'll meet it. The nav bar order (Shifts → Staff → Skills → Leaves → Profiles → Generate → Rosters) roughly mirrors setup order.

Shift group — e.g. DSG, NSG, "Off", "Leaves"
A category of shifts with two flags that the solver cares about deeply: is work shift (does it count as working time?) and is night shift (used for rest rules and fairness). Non-work groups exist too — leave and off-duty codes live in a non-work group.
Shift — e.g. D078 "0700–1500"
A daily template: a short code, a start and end time, and net work minutes (duration minus breaks). Times are stored as minutes from midnight (420 = 07:00). A shift whose end is ≤ its start crosses midnight — that's how night shifts are encoded, not a data error.
Skill — a type with values
A key–value tag, e.g. type seniority with values senior/junior. Attach values to staff; a demand can then require one, and only matching staff count toward it. Matching is exact — no hierarchy, "senior" does not satisfy "junior".
Staff
A person, in a staff group, with an employee ID, optional skills, and optional permitted shifts. If a staff member has no permitted-shift entries, they're allowed on every shift — an empty list means "unrestricted", not "nothing". Staff are soft-deleted (delete → restore is possible).
Leave
A dated absence for one staff member, carrying a shift code (default AL). During solving, a leave becomes a locked assignment of that code — the person can't work that day. See Gotcha 2.
Profile — the central object
A reusable scenario: which shifts and which staff are in play (its Shifts and Staff tabs), plus the solver's Config (fairness weights, time limit) and Rules (conditional constraints). Think "Ward A with its usual team and policies". Staff can be temporarily excluded from a profile without being removed. Profiles can be duplicated.
Demand
"On this date, between these hours, I need at least N people (optionally with skill X)." Demands are the only thing that makes anyone work: a day with no demand is a day nobody is scheduled.
Roster
One solver run over a profile + date range + demands. Status lifecycle: running → draft or failed; a draft you accept becomes approved. Discarding deletes a non-approved roster.
Target hours per staff
Set at generation time: the number of working hours each staff member must accumulate over the roster window. This is an exact-equality rule — see Gotcha 1, it's the #1 cause of failures.
Chaining / Regenerate
A roster can name a previous roster; the solver then carries state across the boundary — consecutive-workday counts, and who ended the last roster on a night shift (they're still asleep on day 1, and their overnight hours count toward early-morning coverage). The UI's Regenerate action reuses a roster's demands to re-run it.

4The workflow, start to finish

1 Shift groups & shifts→ 2 Skills→ 3 Staff→ 4 Profile→ 5 Generate→ 6 Review & approve
  1. Shifts page. Create shift groups first (a work group like DSG; a night group with the night flag; one non-work group for leave/off codes), then the shifts inside them. Create an AL (leave) shift in the non-work group — you will need it (Gotcha 2).
  2. Skills page (optional). Only needed if demands will require specific capabilities.
  3. Staff page. Create staff groups and people; attach skills; restrict permitted shifts only where someone genuinely can't work certain shifts.
  4. Profiles page. Create a profile; on its tabs add the shifts (including AL) and staff, review Config weights, and add Rules if you have policies like "no work the day after a night shift".
  5. Generate page — a 4-step wizard: pick profile & name → date range & target hours per staff → preview which leaves will apply → enter demands per day. Submitting dispatches the solver.
  6. Rosters page. A calendar of drafts (beige) and approved rosters (green). Open the draft, inspect the timetable, then approve, discard, or regenerate. Export to Excel is available on the roster view.
Rough capacity check before you generate: total demand headcount × shift length must be coverable by (number of staff × target hours), and each person's target must be reachable as a sum of whole shifts. If the arithmetic can't work, the solver will spend the whole time limit proving it can't work.

5How the solver thinks

Knowing its hard rules explains almost every "why did it do that".

Hard rules (never violated)

  • At most one work shift per person per day.
  • Every person works exactly their target hours over the window — not "up to".
  • Every demand window is covered by enough (skill-matching) people at every point in time. Overnight shifts from the previous day count toward early-morning coverage.
  • People only get shifts they're permitted.
  • Leaves are locked in; no work that day for that person.
  • No demand on a day ⇒ no work shifts on that day, for anyone.
  • Your conditional Rules, read as: if staff did [trigger group] on day D, then on day D + offset, [enforce group] must equal [value]. * is a wildcard. Example: trigger NSG = 1, offset 1, enforce * = 0 → "no shift the day after a night shift".

Soft preferences (traded off by weight)

Among all schedules satisfying the hard rules, it minimises a weighted mix — the weights are per-profile on the Config tab:

WeightDefaultPenalises
weight_consec100Long consecutive-workday streaks
weight_overstaff20Uneven over-staffing across days (spread, not total)
weight_burden10One person carrying an outsized night/weekend load…
weight_night2…where each night shift costs this much burden
weight_weekend1…and each weekend day this much
time_limit600 sWall-clock budget per solve (see Gotcha 5)

Result statuses inside a draft: OPTIMAL means provably best under these weights; FEASIBLE means valid but the time limit expired before optimality was proven — both are usable schedules.

6Gotchas

1Target hours are exact — the top cause of "failed"

Each person must work precisely the target. With only 8-hour shifts, a 36-hour target is impossible for everyone and the whole roster fails after burning the full time limit. Make the target a reachable sum of your shift lengths, and check total supply ≈ total demand.

2Leave codes must exist as shifts in the profile

A leave stores a shift code (default AL). When solving, that code must resolve to a shift included in the profile. If it doesn't, the run errors out — the roster turns failed with "Shift 'AL' not found". Fix: create the leave shift in a non-work group and add it to every profile's Shifts tab.

3No demand = nobody works, and target hours still apply

Days without a demand are force-blocked for everyone. So if your window is 14 days but you entered demands for 10, all target hours must fit into those 10 days — a common hidden cause of Gotcha 1. Enter a demand for every day people should work, even a minimal one.

4A roster can get stuck in running forever

If Redis or the worker is down when a roster is submitted (or the worker dies mid-solve), nothing ever flips it to failed — the UI polls indefinitely. Check logs -f celery_worker; if the run is truly dead, Discard the roster and re-generate.

5"Nothing is happening" usually means the solver is just working

The default time budget is 10 minutes per roster, and an infeasible problem uses all of it before failing. For small teams, set time_limit to 30–60 (seconds) in the profile Config so experiments fail fast. Also note the worker solves one roster at a time — a second submission queues behind the first.

6Empty permitted-shifts means all shifts

Counter-intuitive but by design: a person with no permitted-shift entries can be assigned any shift in the profile, including nights. The moment you add one entry, they're restricted to exactly the listed shifts.

7Delete is forever — and works on approved rosters

Discard refuses to touch approved rosters, but Delete hard-deletes anything, approved included, with no undo. That's intentional (the UI offers it) — just be sure. The same applies to down -v, which erases the entire database.

8Overnight shifts: end before start is correct

A shift like N2210 stored as 22:00 → 06:00 (end < start) is the night-shift convention, not bad data. The solver extends it into the next day, counts it toward next-morning coverage, and the timetable view renders both segments.

9Skill matching is exact and demands with skills are additive

A demand requiring seniority: senior is only satisfied by staff with exactly that value. And it's counted separately from the same window's unskilled demand — the fairness "overstaffing" measure only tracks the unskilled headcount, so skilled demands can make staffing look uneven; that's expected.

10Dev quirk: solver code changes need a worker restart

The API hot-reloads on file changes; the celery worker doesn't. If anyone edits backend/solver/ or backend/app/worker/: docker compose -f docker-compose.dev.yml restart celery_worker, or the old code keeps running.

7Troubleshooting

SymptomLikely cause → fix
Roster failed, error "No feasible solution found within time limit"Constraints are contradictory — almost always Gotchas 1 or 3. Recheck target hours vs. shift lengths and demand-less days.
Roster failed, error "Shift 'X' not found" / "Staff 'X' not found"A leave references a shift code missing from the profile (Gotcha 2), or leave data points at staff outside it.
Stuck on running > time limitWorker or Redis down (Gotcha 4). ps then logs -f celery_worker; discard and re-run.
Everything logs you out / 401sJWT expired (24 h). Sign in again.
Frontend blank or API unreachabledocker compose … ps — if a container is down, up -d; check port collisions (5173, 8000, 8001, 5432, 6379) with other projects, including the stale stack in /home/guo/roster-engine.
"relation does not exist" errors from the APIMigrations not applied (fresh DB): exec backend alembic upgrade head.
Complete reset wanteddown -v, up -d, migrate, re-create the admin user (§2). Everything else was UI-entered and is gone.

8The MCP server

The mcp container (port 8001, endpoint http://localhost:8001/mcp) exposes the roster API as MCP tools — list_profiles, create_roster, get_roster_status, and friends — so an MCP-capable chatbot (Claude Code, OpenClaw, …) can generate rosters conversationally. It authenticates to the backend with its own service account (mcp_service, password in .env). Full client-setup walkthrough: MCP-guide.md in the repo root. You can ignore this service entirely if you only use the web app.
